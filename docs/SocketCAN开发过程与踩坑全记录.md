## 一、你的目标场景

在你的**工业物联网数据采集网关**里，CAN总线承担**内部转发**角色：

```bash
Modbus RTU采集 → 网关内部 → CAN总线转发 → 网关内部 → MQTT上云
```

你不是让CAN去接外部设备，而是把CAN当成网关内部不同模块之间的**数据总线**。采集线程把传感器数据打包成CAN帧发出去，上云线程从CAN总线上收到帧后打包成JSON发给华为云。

## 二、你用的工具和接口

- **内核接口**：SocketCAN（Linux内核原生支持，把CAN设备当成网络套接字操作）

- **命令行工具**：`candump`（看CAN总线上的帧）、`cansend`（手动发帧）

- **编程接口**：C语言的 `socket()`、`bind()`、`write()`、`read()`，搭配 `struct can_frame` 结构体

- **虚拟设备**：`vcan0`（你当时没有真实CAN硬件，用虚拟CAN接口开发调试）

创建虚拟CAN接口的命令：

```bash
sudo ip link add dev vcan0 type vcan
sudo ip link set up vcan0
```

## 三、开发过程（你走过的路）

### 第1步：理解SocketCAN的"公告板"模型

这是你最重要的心智突破。你给自己总结的模型是：

> CAN总线像一块公告板。你贴公告上去（`write`），别人看公告（`read`）。不是你发给特定的人，是别人自己决定要不要听。
> 
> 

技术上对应：

- CAN帧没有"源地址"和"目的地址"，只有**ID**

- 每个节点根据ID判断"这条消息我需不需要处理"

- 你网关里两个线程，一个只写（采集线程），一个只读（上云线程），通过CAN ID约定数据含义

### 第2步：写出第一版收发代码

你按照SocketCAN的标准写法，用 `socket(PF_CAN, SOCK_RAW, CAN_RAW)` 创建套接字，然后 `bind` 到 `vcan0`，采集线程里封装 `struct can_frame` 后 `write`，上云线程里 `read`。

编译命令大致是这样：

```bash
gcc -o can_test can_test.c
```

### 第3步：用candump验证——发现收不到自己发的帧

这是你踩的第一个大坑。

你在采集线程里 `write` 了一个CAN帧，然后在**同一个程序的上云线程**里 `read`，结果读不到。

你开另一个终端跑 `candump vcan0`，能看到帧出现在总线上。说明**帧确实发出去了，是你自己的程序收不到**。

## 四、核心踩坑：CAN_RAW_RECV_OWN_MSGS

### 问题本质：

SocketCAN默认行为是：**一个套接字发出去的帧，不会回送给同一个套接字**。这是内核层的过滤机制，不是你的代码写错了。

你当时的情况是：采集线程和上云线程共用了**同一个套接字**（或者各自创建了套接字但绑定了同一个接口，而默认设置导致自收不发）。

### 你的解决过程：

1. 先用 `candump` 确认帧确实在总线上 → 排除硬件/驱动问题

2. 查Linux内核文档，找到`CAN_RAW_RECV_OWN_MSGS` 这个套接字选项

3. 在创建套接字后、bind之前，加上这段关键代码：

```c
int sock = socket(PF_CAN, SOCK_RAW, CAN_RAW);
int recv_own_msgs = 1;  // 1 = 开启自接收
setsockopt(sock, SOL_CAN_RAW, CAN_RAW_RECV_OWN_MSGS,
           &recv_own_msgs, sizeof(recv_own_msgs));
```

### 你的"终于懂了"时刻（5月9日）：

> CAN自收发问题终于解决。你发现是SocketCAN默认不接收自己发送的帧，需要设置`CAN_RAW_RECV_OWN_MSGS`。你说"终于懂了"。
> 
> 

## 五、后续你做的工程化工作

踩完这个坑之后，你把CAN模块封装进了网关的完整架构里：

1. **CAN帧格式约定**：CAN ID 0x100代表温度，0x101代表湿度，0x102代表气压；数据段用4字节float

2. **多线程分离**：采集线程只写CAN，上云线程只读CAN，两者通过CAN总线解耦

3. **互斥锁保护**：虽然CAN收发本身是内核安全的，但你的共享数据区（最新传感器值）用了互斥锁保护

4. **全链路打通**（5月10日）：Modbus → 采集线程写CAN → 上云线程读CAN → MQTT上云，完整跑通
